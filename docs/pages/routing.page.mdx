By default `vite-plugin-ssr` does:
 - Server-side Routing, but we can also use Client-side Routing instead.
 - Filesystem Routing, but we can also use Route Strings and Route Functions instead.

## Server-side Routing or Client-side Routing?

In general, we recommand using Server-side Routing (full HTML reload upon page navigation) instead of Client-side Routing (DOM mutations upon page navigation).

That said, `vite-plugin-ssr` hast first-class support for Client-side Routing.

More infos at [Server-side Routing VS Client-side Routing](/SR-vs-CR).

## Filesystem Routing VS Route Strings VS Route Functions

By default, `vite-plugin-ssr` does Filesystem Routing:

```
FILESYSTEM                  URL
pages/index.page.js         /
pages/about.page.js         /about
pages/faq/index.page.js     /faq
```

For advanced routes we define so-called "Route Strings" and "Route Functions" in `.page.route.js` files.

```js
// /pages/product.page.route.js

// Route Strings gives us an easy way to define parameterized routes.
export default '/product/:productId'
// `productId` is available at `pageContext.routeParams.productId`.
```

```js
// /pages/admin.page.route.js

// Route Functions gives us full programmatic flexibility to implement precise routing
// such as route guards.
export default (pageContext) => {
  const { url } = pageContext
  if (url === '/admin' && pageContext.user.isAdmin) {
    return true
  }
}
```
